## 2. DIAGNOSTIC INITIAL — ROOT CAUSE ANALYSIS

### 2.1 Symptôme rapporté

L'opérateur clique "Build Agent" dans `http://localhost:3001/factory` → **build TOUJOURS en échec**.

Logs serveur observés :
```
[ERROR] [Production Readiness Gate] FAILURE: Production BLOCKED:
  [STRICT] Only 1 Groq API key — no rotation fallback available;
  [STRICT] Twenty custom fields enabled but not strictly enforced — may silently skip
```

### 2.2 Cause racine identifiée — DEUX problèmes entrelacés

#### Problème A : `FACTORY_BUILD_STRICT=true` transforme les warnings en blockers

**Fichier** : `server/src/factory/readiness-gate.ts`, lignes 82-88

```typescript
if (config.factory?.buildStrict) {
  const warningChecks = checks.filter((c) => c.status === "warn");
  for (const wc of warningChecks) {
    blockers.push(`[STRICT] ${wc.message}`);  // ← Warning → Blocker
  }
  warnings = [];
}
```

Avec `buildStrict=true` (défaut dans `.env.example`), les deux warnings suivants deviennent des **blockers fatals** :

| Check | Source | Status normal | Avec Strict |
|-------|--------|--------------|-------------|
| `checkLlmConfiguration()` L144-149 | `keyCount < 2` | `warn` | **BLOCKED** |
| `checkCustomFieldsPolicy()` L395-399 | `!config.crm.strict.customFields` | `warn` | **BLOCKED** |

#### Problème B : L'UI envoie des secrets masqués dans le body du build

**Fichier** : `server/src/routes/factory-ui.routes.ts`, fonction `startBuild()` (ancienne version)

**Séquence problématique** :
1. `init()` appelle `GET /api/factory/config` → retourne config avec secrets **redacted** via `redactSecrets()`
   - `redactSecrets()` (factory.routes.ts L624-668) transforme les clés en `gsk_t04o...XDf`
2. `currentConfig` stocke cette config redactée côté client
3. `collectConfig()` clone `currentConfig` via `JSON.parse(JSON.stringify(currentConfig))`
4. Pour les Groq keys : le textarea contient les valeurs redactées (avec `...`)
   - Le check `!llmGroqKeysRaw.includes('...')` est FALSE → le `else if` ne s'exécute pas
   - Le `else` implicite garde `currentConfig.llm.groq.apiKeys` = array de strings redactés
5. `startBuild()` envoie `POST /api/factory/build { config }` avec les secrets redactés
6. `executeBuildPipeline(config)` :
   - Step 3 : `saveConfig(config)` détecte les valeurs redactées et les **patche** depuis le backup .env ✅
   - Step 8 : `runReadinessChecks(config)` utilise l'objet config **original** (avec valeurs redactées)
   - Les clés redactées sont des strings non-vides → `apiKeys.length` = nombre correct
   - **MAIS** si le .env actuel n'a qu'1 clé Groq (pas 6), le config reflète ça

**Conclusion** : Le problème B n'est pas toujours la cause directe du "1 Groq key", mais le flow est fragile et non-déterministe. Le vrai fix est de **ne jamais envoyer le config dans le body du build**.

### 2.3 Solution architecturale appliquée

**Nouveau flow dans `startBuild()` (version corrigée)** :

```
1. collectConfig() → config object (avec secrets potentiellement redactés)
2. PUT /api/factory/config { config } → saveConfig() patche les secrets, écrit .env
3. POST /api/factory/build {} → le serveur appelle loadCurrentConfig() qui lit le .env RÉEL
4. executeBuildPipeline(configFromEnv) → tous les secrets sont réels, pas redactés
```

Ceci est exactement ce que le smoke test fait, et c'est prouvé fonctionnel.

#### Nouvelle fonction JS — `getRemediationHint(blocker)` (lignes ~1551-1565)

```javascript
function getRemediationHint(blocker) {
  var b = blocker.toLowerCase();
  if (b.includes('groq api key') && b.includes('1'))
    return 'Ajoutez une 2e cle Groq dans API Secrets > Groq LLM. Ou desactivez le Mode Strict.';
  if (b.includes('custom fields') && b.includes('not strictly'))
    return 'Activez Strict Custom Fields dans CRM Engine > Strict Mode. Ou desactivez Mode Strict.';
  if (b.includes('database'))
    return 'Verifiez PostgreSQL et DATABASE_URL dans API Secrets > Database.';
  if (b.includes('api url') || b.includes('api key missing'))
    return 'Remplissez les identifiants CRM dans API Secrets. Ou mettez CRM sur "None".';
  if (b.includes('kill switch'))
    return 'Desactivez le Kill Switch dans LLM Provider > Factory Build Controls.';
  if (b.includes('agent name') || b.includes('agency name'))
    return 'Remplissez le nom dans Identity.';
  if (b.includes('[strict]'))
    return 'Avertissement devenu bloquant a cause du Mode Strict. Corrigez-le ou desactivez Strict.';
  return '';
}
```

#### Changement JS — `startBuild()` (lignes ~1568-1658) — LE FIX PRINCIPAL

**Avant** (flow problématique) :
```javascript
async function startBuild() {
  const config = collectConfig();
  const data = await api('/build', { method: 'POST', body: JSON.stringify({ config }) });
  // ↑ Envoie config avec secrets redactés → build échoue
}
```

**Après** (flow corrigé en 2 étapes) :
```javascript
async function startBuild() {
  // STEP 1: Sauvegarder le config dans .env
  // saveConfig() côté serveur patche automatiquement les valeurs redactées
  var config = collectConfig();
  await api('/config', { method: 'PUT', body: JSON.stringify({ config: config }) });

  // STEP 2: Lancer le build avec body VIDE
  // Le serveur appelle loadCurrentConfig() qui lit les vrais secrets depuis .env
  var data = await api('/build', { method: 'POST', body: JSON.stringify({}) });

  // STEP 3: Afficher résultat
  if (data.success) {
    // Banner vert + embed code + iframe URL
    banner.className = 'build-banner success';
    var embedUrl = window.location.origin + '/embed?widget_id=default';
    var snippet = '<iframe src="' + embedUrl + '" ...></iframe>';
    document.getElementById('embedCode').textContent = snippet;
    embedResult.style.display = 'block';
  } else {
    // Banner rouge + erreurs + hints de remédiation via getRemediationHint()
    banner.className = 'build-banner failure';
    // ... render errors with fix hints
  }
}
```

### P0 — BLOQUANTS PRODUCTION (à faire maintenant)

#### P0.1 — Test du build depuis l'UI réelle avec CRM=twenty + Strict=true

Le smoke test prouve le flow **programmatique** avec CRM=none et Strict=false.
Il faut maintenant tester le scénario complet :
1. Ouvrir `http://localhost:3001/factory` dans un navigateur

**Pourquoi c'est P0** : Le flow save-then-build (`PUT /config` puis `POST /build {}`) est prouvé côté script, mais pas encore validé via le navigateur réel. Il y a un risque que `collectConfig()` produise un objet config incomplet ou que `saveConfig()` échoue sur un edge case de patching de secrets.

**Action** : Ouvrir le navigateur, faire le test manuellement, noter le résultat. Si ça échoue, lire les logs serveur (stdout) et les factory logs (`GET /api/factory/logs`).

#### P0.2 — Le `PUT /config` peut perdre des clés non modélisées

`saveConfig(config)` appelle `agentConfigToEnv(config)` qui génère un `.env` **complet** à partir de l'objet `AgentConfig`. Mais certaines clés du `.env.example` ne sont PAS dans le modèle `AgentConfig` :

```
CHAT_DEBUG=1
CHAT_DEBUG_MAX_STR=1500
COMPANY_TAGLINE=...
COMPANY_DESCRIPTION=...
COMPANY_SERVICES=...
TARGET_AUDIENCE=...
OPENROUTER_APP_URL=...
CRM_AUTO_DETECT_PHONE_COUNTRY=...
CRM_NOTES_DATE_FORMAT=...
CRM_EXPERIMENTAL_FEATURES=...
CRM_CACHE_TTL_SECONDS=...
CRM_BATCH_*=...
SLACK_WEBHOOK_URL=...
```

Ces clés sont perdues lors du `saveConfig()` car `agentConfigToEnv()` ne les émet pas. Le mécanisme d'append des secrets préservés (L847-862 de config-synthesizer.ts) ne couvre que les clés identifiées comme `isSecretKey()`. Les clés non-secret non-modélisées sont **silencieusement supprimées**.

**Action** : Ajouter dans `saveConfig()` un mécanisme de "passthrough" qui préserve toute clé du `.env` actuel qui n'est pas dans la sortie de `agentConfigToEnv()`. Ou étendre `AgentConfig` / `agentConfigToEnv()` pour couvrir toutes les clés.

#### P0.3 — Sécurité : `.env.example` contient des clés API réelles

Le fichier `.env.example` contient des clés Groq, OpenRouter, Twenty, JWT et admin **en clair**. Si le repo est poussé sur un dépôt public, toutes ces clés sont compromises.

**Action** : Remplacer les valeurs réelles par des placeholders dans `.env.example` et stocker les vraies clés dans un `.env.local` ignoré par git. Ou utiliser un vault (1Password CLI, Doppler, etc.).

### 7.5 Pièges connus

1. **Ne jamais envoyer un config body avec des secrets redactés à `POST /build`**. Toujours `{}`.
2. **`saveConfig()` crée des backups `.env.backup.<timestamp>`**. Il y a un max de 10 (`MAX_BACKUP_COUNT`). Les vieux sont purgés par `rotateBackups()`.
3. **`loadCurrentConfig()` merge `process.env` + contenu du fichier `.env`**. Le fichier `.env` a priorité (override). Mais si le serveur a été démarré avec d'anciens env vars, `process.env` peut contenir des valeurs stale.
4. **Les checks de readiness sont async** (check 1: DB, check 15: catalog count). Les autres sont sync.
5. **Le `logBuffer` est un ring buffer de 500 entrées max**. Les logs anciens sont perdus. Pas de persistance DB.
6. **Le rate limiter utilise PostgreSQL** (`createPostgresStore()`). Si la DB est down, le rate limiter peut aussi fail.
